# Audit storylets with requires_flag for missing sets_flag patches

## Context

T-1777564800228 found that `glenn_pastime_paradise.head_to_evening` was
missing its migration-applied `sets_flag: ["glenn_gave_direction"]` in the
live DB. The mechanism that dropped it (suspected Content Studio replace)
is tracked separately in T-1777574470935.

This ticket is the reactive audit: find every storylet currently in the
database with a `requirements.requires_flag` predicate, and verify that
some choice somewhere actually writes that flag via `sets_flag`. Any flag
that is *required* but never *set* is dead content — the gated storylet is
unreachable, which is exactly the failure mode T-228 just fixed for one
storylet.

## Acceptance criteria

- [ ] Query for all storylets with `requirements ->> 'requires_flag'` set,
      collect the flag names
- [ ] For each required flag, search choice_log fixtures or storylet
      `choices[*].sets_flag` and `nodes[*].micro_choices[*].sets_flag` for
      a writer
- [ ] Report: list of (required_flag, storylets_requiring_it,
      writer_storylet+choice_id_or_null)
- [ ] For any flag with no writer, file a fix migration to add `sets_flag`
      to the appropriate choice (or remove the orphaned `requires_flag` if
      the gate is stale)
- [ ] Bonus: extend `scripts/validate-content.ts` (or similar) to run this
      check at content-load time and warn on dead gates

## Out of scope

- Fixing the underlying clobber mechanism (T-1777574470935)
- Auditing OTHER choice-level fields for drops (e.g. `precludes`,
  `practices_skills`) — separate audits if T-1777574470935 confirms a
  systemic issue

## Files of interest

- Storylets table — query for `requirements ? 'requires_flag'`
- `scripts/validate-content.ts` — content-validation entry point
